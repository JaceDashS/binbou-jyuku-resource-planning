# 04. QR / 出席 / 出退勤ポリシー

## 1. 文書の目的

この文書は、**QRの種類、表示/スキャンモード、出席QR生成ルール、teacherの出勤/退勤ルール、studentの出席ルール、出席noteルール**を整理した基準文書である。  
サーバー接続用認証とトークン詳細は `02-runtime-architecture-and-connection.md`、権限/UI導線構造は `03-role-permission-and-client-flow.md`、保存構造は `05-domain-rules-and-schema.md` で詳細に定義する。

---

## 2. QRの種類区分

**何のためのQRか（用途）** と **どのように使うか（使用モード）** を分けて説明する。

要点は次のとおりである。

- **サーバー接続QR**: 新しいteacherクライアントアプリを **マスターサーバーに接続し、初回ログインに必要なサーバー情報と接続トークンを確保** するためのQR
- **出席QR**: studentまたはteacherの **アカウント情報を表示して出席処理** を行うためのQR
- **表示モード**: 自分のQRを画面に表示するモード
- **スキャンモード**: 他人のQRを読み取るモード

つまり、**サーバー接続QR / 出席QRは用途の区分** であり、**表示モード / スキャンモードは使用方式の区分** である。

---

## 3. 用途区分

### 3-1. サーバー接続QR

- 目的: **teacherクライアントがマスターサーバーのアドレスを受け取り、初回ログインに必要なサーバー情報と接続トークンを確保** するためのQR
- 含まれる情報: マスターサーバー Private IP + ポート (`40343`) + 実行時に生成された接続トークン
- 表示位置: **すでにログイン済みのteacherクライアントまたはマスターアプリ画面**

例示構造:

```text
SERVER:http://192.168.x.x:40343
TOKEN:random-generated-token
````

実際の利用シナリオ例:

```text
状況: 新しいタブレット1台をteacher用端末として学習塾に追加する

1. マスターアプリの起動時にランダム接続トークンが生成される
2. マスターアプリ画面、またはすでにログイン済みのteacherアプリ画面でサーバー接続QRを表示する
3. 新しいタブレットでteacherクライアントを起動する
4. 新しいタブレットがサーバー接続QRをスキャンする
5. アプリがマスターサーバーのアドレスと接続トークンを自動入力する
6. teacherが login_id / password を入力してログインする
7. ログイン成功後、teacher_auth_token が発行され、機能を利用できる
```

このQRは **出席処理用ではなく、teacher端末のサーバーアドレス + 接続トークン入力用** である。

### 3-2. 出席QR

#### 基本概念

* 目的: **studentまたはteacherのアカウント情報をQRで表示して出席処理** を行うためのQR
* teacher QRも本人のアカウント情報をQRで表示できる
* 出席QRはサーバー接続QRとは別概念であり、役割を混同しないようUIで明確に区別する必要がある
* student QRとteacher QRの区分は別途typeフィールドを置かず、**code prefix体系** で処理する
* たとえば `student_code` は `S` で始まり、`teacher_code` は `T` で始まる

#### 出席QR生成形式

出席QRは次の形式を使用する。

```text
LOGIN_ID|CODE|TIME_SLOT|PORT
```

例:

```text
kim01|S0001|29384721|40343
park02|T0003|29384721|40343
```

構成要素の意味:

* `LOGIN_ID`: 対象アカウントの login_id
* `CODE`: student_code または teacher_code
* `TIME_SLOT`: 60秒単位の再生成基準値
* `PORT`: 現在アプリで使用しているポート番号

#### student出席QR生成原則

* student端末は初回ログイン時にQR生成に必要なローカル保存情報を確保する
* 以後、student出席QRは **オフラインでローカル生成** できる
* student出席QRは **`login_id + student_code + time_slot + port`** 構造で生成する
* `TIME_SLOT` は **60秒単位の再生成基準値** として使用する
* したがってstudentアプリは、同じネットワークに継続接続されていなくてもQRを継続的に再生成できる

#### teacher出席QR生成原則

* teacher端末は初回ログイン時に teacher_code およびQR生成に必要なローカル保存情報を確保する
* 以後、teacher出席QRは **現在のログイン状態と無関係に** ローカルで生成できる
* teacher出席QRは **`login_id + teacher_code + time_slot + port`** 構造で生成する
* `TIME_SLOT` は **60秒単位の再生成基準値** として使用する
* したがってteacherも現在 `teacher_auth_token` が期限切れ状態であっても、過去に必要情報が保存されていれば本人QRを継続表示できる

#### 検証およびセキュリティルール

* スキャナーは `CODE` prefix を基準に student QR と teacher QR を区別する
* 出席QRの再生成周期は **60秒単位** とする
* サーバーは **現在のマスターサーバーのローカル time_slot と QR の time_slot が正確に一致する場合のみ有効** として扱う
* サーバーはQR検証時に次の順序に従う

1. QRのパース
2. code prefixで student / teacher を区分
3. 対象テーブルで code を検索
4. rowの `login_id` とQRの `login_id` が一致するか確認
5. QRの `time_slot` と現在のマスターサーバー `time_slot` が一致するか確認
6. `port` 値が現在のサーバーポートと一致するか確認
7. すべて一致した場合、有効として処理

* 現在のQR生成ルールは **MVP 1次向けの単純検証構造** であり、強い偽造防止を保証する秘密署名構造ではない
* ポート番号はQR生成ルールの構成要素として使用するが、強い秘密値とはみなさない
* 後続バージョンでは、seed、HMAC、署名ベース方式によるQR偽造防止強化を検討できる

#### 学生アカウントおよび端末原則

* 学生は別途の会員登録UIを使用しない
* 学生アカウントは teacher または admin が登録する
* 学生端末では **初回ログイン1回と初回QR生成1回** のみマスターと通信可能な状態が必要である
* 初回ログインと初回QR生成以後は、学生が同じネットワークに継続接続されていなくても基本的なQR表示利用は可能である
* 出席QR自体にはWiFi情報を含めない

#### teacher端末使用原則

* teacher QR表示は、過去に一度ログインして必要なローカル保存情報を確保した端末であれば、現在のログイン状態と無関係に可能である
* 一方、出席処理、チェックイン、チェックアウトのためのスキャンモードは **ログイン済みteacher端末** でのみ可能である
* つまり、QRを見せることと、QRを読み取ってサーバーに記録することは区別する
* 出席処理、チェックイン、チェックアウト時に **接続トークンは不要** である
* 接続トークンは teacher端末がマスターサーバーに初回ログインするときにのみ使用する
* `teacher_auth_token` が無効化された場合、teacherは再ログイン後にスキャンモードとサーバー記録機能を利用できる

実際の利用シナリオ例 1: 学生アカウント登録および初回QR生成

```text
状況: 新しい学生をシステムに登録する

1. teacher または admin がマスターで学生アカウントを登録する
2. 学生端末で初回ログインを1回実行する
3. マスターと通信して初回QR生成に必要な情報を保存する
4. 以後、学生はオフラインでも本人QR画面を継続利用できる
```

実際の利用シナリオ例 2: 学生出席

```text
状況: 学生が学習塾に到着する

1. 学生が自分のアプリで本人QRを画面に表示する
2. ログイン済みteacher端末がスキャンモードで学生QRを読み取る
3. システムが login_id, student_code, time_slot, port を検証する
4. 有効であれば学生出席が記録される
```

実際の利用シナリオ例 3: teacher出勤

```text
状況: teacherが出勤する

1. teacherが本人アプリで本人QRを画面に表示する
2. 別のログイン済みteacher端末のスキャンモードがそのQRを読み取る
3. システムが login_id, teacher_code, time_slot, port を検証する
4. 同日の初回スキャンであれば出勤として処理される
```

実際の利用シナリオ例 4: teacher退勤

```text
状況: teacherが退勤する

1. teacherが自分のアプリで再度本人QRを画面に表示する
2. 別のログイン済みteacher端末のスキャンモードがそのQRを読み取る
3. システムが同日の既存出勤記録を確認する
4. 2回目のスキャンであれば退勤として処理される
```

つまり、出席QRは **サーバー接続用ではなく、人の出席処理用** である。

## 4. 使用モード区分

### 4-1. 表示モード

* **studentとteacherの両方が使用可能**
* 表示モードは **自分のQRを画面に表示するモード** である
* studentは表示モードで **本人の出席用QR** を画面に表示する
* teacherも表示モードで **本人QR** を画面に表示できる
* teacher QRは別のログイン済みteacher端末のスキャンモードで読み取られ、**出勤/退勤処理** に使用できる

実際の利用シナリオ例:

```text
学生アプリ起動
→ 本人QR表示
→ teacherがスキャン
→ 学生出席処理
```

または

```text
teacherアプリ起動
→ 本人QR表示
→ 別のログイン済みteacher端末のスキャンモードが読み取る
→ teacher出勤または退勤処理
```

### 4-2. スキャンモード

* **スキャンモードはteacherのみが使用できる機能** である
* スキャンモードは **他人のQRを読み取るモード** である
* スキャンモードが有効化されたteacherアプリでQRをスキャンすると出席処理が行われる
* student QRをスキャンすると student 出席処理が実行される
* teacher QRをスキャンすると teacher 出勤/退勤処理が実行される
* スキャンモードは **ログイン済みteacher端末でのみ使用可能** であり、`teacher_auth_token` が無効であれば使用できない

実際の利用シナリオ例:

```text
teacherがスキャンモードを起動
→ 学生QRをスキャン
→ 学生出席記録
```

または

```text
teacherがスキャンモードを起動
→ teacher QRをスキャン
→ teacher出勤または退勤記録
```

## 5. 混同しないための一文整理

```text
サーバー接続QR = 端末をサーバーに接続し、初回ログインに必要なサーバー情報と接続トークンを確保するためのQR
出席QR = 人の出席を処理するためのQR
表示モード = 自分のQRを見せるモード
スキャンモード = 他人のQRを読み取るモード
```

## 6. 出席処理方式

### 6-1. 共通原則

* 現場ではQRコード等を使って出席処理が可能
* 出席機能もクライアントアプリ内で提供する
* QR関連機能は **表示モード** と **スキャンモード** に区分する
* **表示モード** は student と teacher の両方が使用でき、本人アカウントQRを画面に表示する用途である
* **スキャンモード** はログイン済みteacher端末でのみ使用でき、スキャンモードが有効化されたアプリでQRを読み取ると出席処理が実行される
* スキャンモードでstudent QRを読み取るとstudent出席処理を実行する
* スキャンモードでteacher QRを読み取るとteacher出勤/退勤処理を実行する
* 出席処理、チェックイン、チェックアウト時に **接続トークンは再使用しない**

### 6-2. teacher出退勤ルール

* teacher QRスキャンルールは **同日の初回スキャンは出勤**、**2回目のスキャンは退勤**、**3回目以降はエラー処理** と定義する
* つまり、`teacher_attendance` は **1日最大2回のスキャンのみ許可** する
* teacher QRの初回スキャン時に `teacher_attendance` row を生成し、2回目のスキャン時に同じ row の `checkout_at` を更新する
* 出勤のみ記録して退勤を記録しないまま日付が変わった場合、翌日の初回スキャンは **新しい出勤処理** として開始する
* 前日の退勤未記録件は維持し、その後UIで **退勤未記録エラー(`checkout_at IS NULL`)** として表示できるようにする

### 6-3. 学生出席ルール

* student出席は重複入力を許可するが、**同一学生は5分以内に再スキャンできないよう制限** して多重入力を防止する
* 学生が複数回出席できる理由は、連続していない時間帯に複数の授業を受けることがあるためである
* ただしこのルールはMVP 1次の単純保護ルールであり、後続バージョンでは授業/時間ベースでもっと精緻に改善する余地がある

### 6-4. 学生端末接続原則

* 学生は別途の会員登録UIを使用しない
* 学生アカウントは teacher または admin が登録する
* 学生端末では初回ログイン1回と初回QR生成1回のみマスターと通信可能な状態が必要である
* 初回QR生成以後は、学生が同じネットワークに継続接続されていなくても基本的なQR表示利用は可能である

## 7. 出席記録ルール

### 7-1. 基本保存原則

* MVP 1次で `teacher_attendance` と `student_attendance` は **記録自体を保存し、状態は解釈して表示する方式** を使用する
* したがって attendance テーブルには `status` と `reason` を置かず、必要時のみ手動入力用 `note` を使用する
* `teacher_attendance.note`, `student_attendance.note` は基本的にNULL許容
* note は特殊ケースがあったかを簡潔に手動記入する用途である

### 7-2. note作成 / 修正権限

* 出席note関連の作成者IDは **原本作成者** ではなく **現在値の最終修正者** を意味する
* したがって `note_editor_teacher_id` は現在のnote値を最後に修正したteacher IDを保存する
* 原本作成者を確認するには `audit_logs` を参照する必要がある
* non-admin teacher も出席処理過程で note 作成は可能である
* non-admin teacher は自分が最後に修正した出席noteのみ再修正可能であり、他人が最後に修正したnoteは修正できない
* admin は出席note修正に制限を設けない

### 7-3. teacher出退勤解釈ルール

* teacher出退勤状態は保存せず、`checkin_at`, `checkout_at` の値を基準に解釈する
* 同日の初回スキャンは出勤、2回目のスキャンは退勤、3回目以降はエラーとして処理する
* つまり、`teacher_attendance` は1日最大2回のスキャンのみ許可する
* teacher QRの初回スキャンは `teacher_attendance` row 生成、2回目のスキャンは同じ row の `checkout_at` 更新で処理する
* 出勤のみ記録し、退勤を記録しないまま日付が変わった場合、翌日の初回スキャンは新しい出勤として処理する
* 前日の未完了rowは維持し、その後UIで退勤未記録状態として解釈して表示できるようにする

### 7-4. 学生出席ルール

* `student_attendance` はMVP 1次段階で **授業別出席ではなく学生単位スキャン記録** を保存する構造を使用する
* student QRスキャンでは `class_id` を使用しない
* student出席rowは `student_id` と `scanned_at` を中心に保存する
* student出席の日付判定は `scanned_at` のマスターサーバーローカル日付基準で処理する
* 学生は重複出席が可能だが、同一学生基準で5分以内に再スキャンできないよう制限して多重入力を防止する
* 学生が複数回出席できる理由は、連続していない時間帯に複数の授業を受けることがあるためである
* このルールはMVP 1次の単純保護ルールであり、必要時には後続バージョンで授業/時間ベースポリシーへさらに精緻に改善できる
