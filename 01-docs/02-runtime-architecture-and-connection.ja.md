# 02. システム構造 / 接続 / 認証

## 1. 文書目的

この文書は、**マスターアプリ / クライアントアプリ構造、SQLite 運用原則、マスターサーバー実行方式、ネットワーク接続、ポート、接続トークン、teacher 認証トークン、サーバー接続 QR** を整理した基準文書である。

## 2. マスター概念定義

### 2-1. 概念定義

- ここでいうマスターサーバーは内部的には Go ベースのサーバープロセスとして動作するが、ユーザーから複雑に見えないよう **実行ファイル形式のアプリとしてパッケージ提供** する
- ここでいう **マスター** は特定のハードウェア自体ではなく、**Go 言語で実行されるマスターサーバープロセス** を意味する
- つまりシステムの中心は物理機器そのものではなく、**原本 DB を保有し API を提供する Go ベースの実行主体** である
- source of truth はこのマスターサーバーが管理する SQLite DB に存在する
- 他の機器は補助接続端末にすぎず、原本 DB を直接保有したり代替したりはできない

### 2-2. 実行形態および運用範囲

- マスターアプリは運用方式により **前面 UI として実行される場合もあれば、バックグラウンドでサーバープロセスとして維持される場合もある**
- つまり、ユーザーが常にマスターアプリ画面を開いていなくても、マスターサーバープロセスが生きていればシステムの中心役割は維持される
- 1次バージョンではマスターサーバーを他の機器へ移す **公式マイグレーション機能はサポートしない**
- ただし後続バージョンでは、バックアップデータおよび SQLite DB を新しい機器へ移す **マスターマイグレーションオプション** を追加できる
- マスターサーバーが動作する実行環境の要求性能上限は **ミニ PC レベルで安定動作可能な範囲** に設定する

## 3. DB 運用原則

### 3-1. DB 構造原則

- DB は **SQLite ベースのローカル DB** として運用する
- SQLite は **マスターサーバーが実行される環境内部で直接動作** する
- source of truth は **マスターサーバーが管理する SQLite DB** 一つに維持する
- 原本 DB に直接アクセス可能な主体は **マスターサーバーのみ** である
- つまり、システムは複数独立 DB 構造ではなく **単一マスター DB 構造** を前提とする

### 3-2. アクセスおよび運用原則

- 別途クラウド DB や外部サーバー DB は使用しない。本システムは運用コスト最小化を核心目標とするため、今後のバージョンでもクラウド DB や外部サーバー DB を導入する計画は置かない
- 他のアプリや機器は DB ファイルを直接修正しない
- 他機器の読み取り / 書き込み要求はすべてマスターサーバーを通じて処理する
- すべてのデータの最終保存と修正反映はマスターサーバーを通じて実行する
- CSV export 機能は現バージョンでは計画しない。ただし、今後の運用過程で必要性が確認された場合、レポート出力またはバックアップ用 export 機能として追加できる

## 4. マスターアプリとクライアントアプリの配置

### 4-1. 配置原則

- マスターアプリとクライアントアプリは **同じ PC で一緒に動作することも可能**
- つまり、一つの機器内で
  - マスターサーバー実行
  - クライアントアプリ実行  
  が同時に行われる場合がある
- このときマスターアプリは前面 UI として開かれている場合もあれば、**バックグラウンドでサーバープロセスのみ維持される形態** の場合もある
- 逆に、マスターサーバーとクライアントアプリを別の機器に配置することも可能である
- したがって重要なのは「どの機器か」より **どのプロセスがマスターサーバー役割を行うか** にある

### 4-2. 基本構造

システムは二種類のアプリで構成される。

- **マスターアプリ**
- **クライアントアプリ**

マスターアプリは内部で Go ベースのマスターサーバーを実行し、SQLite 原本 DB を管理する。マスターアプリは前面 UI として動作する場合もあれば、必要に応じてバックグラウンドでサーバープロセスを維持する方式で動作する場合もある。

クライアントアプリはマスターサーバーに接続し、QR 画面、出席処理、管理 UI などを使用するユーザーインターフェース役割を担う。

### 4-3. マスターアプリ / クライアントアプリ実行関係

- システムには **マスターアプリ（またはマスターサーバー役割）** と **クライアントアプリ** が存在できる
- マスターアプリは Go サーバーを実行し、SQLite 原本 DB を管理する
- マスターアプリは運用状況により前面 UI 状態の場合もあり、バックグラウンド実行状態の場合もある
- クライアントアプリはマスターサーバーに接続し、QR、出席、管理 UI などを使用する
- この二つは別の機器で動作する場合もあり、**同じ PC で同時動作する場合もある**

### 4-4. 同じ PC で一緒に動作する場合

- 同じ PC でマスターアプリとクライアントアプリが一緒に実行されることがある
- ただし同じ機器であっても、teacher がマスター機能にアクセスする際には `127.0.0.1:40343` と接続トークンを使用したログイン手続きを経ることを原則とする
- つまり、同じ PC で動作しているという理由だけで teacher の初回ログイン手続きを自動的に省略しない

### 4-5. 別の機器で動作する場合

- マスターサーバーとクライアントアプリが別の機器にある場合には、ローカルネットワークアドレスを通じてマスターサーバーに接続する
- この場合でも原本 DB はマスターサーバー側にのみ存在する

## 5. クライアントアプリの localhost 検証

- 同じ PC でマスターアプリとクライアントアプリが一緒に動作することがあるため、クライアントアプリは起動時に `localhost` 検証手続きを行う
- この検証の目的は **同じ機器上でマスターサーバーが実行中か確認すること** である
- ただしこの検証はあくまで **実行有無確認用** であり、teacher がマスターサーバーに実際にリクエストする際には **必ず IP と接続トークン、および teacher_auth_token または teacher アカウント認証情報を使用** しなければならない
- つまり、同じ PC で実行されていても teacher のマスターリクエスト検証手続き自体は省略しない
- 同じ機器からリクエストする場合には `127.0.0.1:40343` と接続トークンを使用できる

例示フロー:

```text
クライアントアプリ起動
→ localhost 確認
→ マスターサーバー実行有無確認
→ teacher が IP 入力
→ 接続トークン入力
→ teacher login_id / password 入力
→ ログイン成功時に teacher_auth_token 発行
→ 以後、機能使用時に Authorization ヘッダーで teacher_auth_token 送信
```

## 6. マスターサーバーポート規則

- マスターサーバーは **固定ポート `40343`** を使用してローカルで実行される
- クライアントアプリは起動時に次のアドレスへ接続を試み、マスターサーバー実行有無を確認する

```text
localhost:40343
```

### 6-1. 同じ PC で実行される場合

```text
クライアントアプリ
→ localhost:40343 接続試行
→ マスターサーバー応答確認
→ サーバーアドレス自動認識
```

### 6-2. 別の機器から接続する場合

別の機器から接続する場合には、マスターサーバーが実行されている機器のローカルネットワーク IP を使用する

```text
http://<master-ip>:40343
```

### 6-3. ポート衝突規則

- ポート衝突を防ぐため、マスターアプリは実行時に **40343 ポート使用可能か先に確認** しなければならない
- すでに当該ポートが使用中であれば、マスターアプリ実行を中断し、ユーザーに案内メッセージを表示する
- **MVP 1次段階では他のポートへ自動切り替えする fallback メカニズムは実装しない**
- マスターサーバーバインド後、ファイアウォール、ローカルネットワークポリシー、OS セキュリティ設定などにより他機器接続が円滑でない場合に対する別途対応は **MVP 1次範囲では考慮しない**
- 外部から入る信号に対するネットワークセキュリティ強化や追加防御ロジックも **MVP 1次では別途考慮せず**、今後の改善可能性のみ開いておく

### 6-4. ネットワーク切断および WiFi 再接続時の処理原則

- 現在のバージョンでは、WiFi が切れた後に再接続される状況に対する **自動 fallback / 自動復旧ロジック** は実装しない
- つまり、アプリがネットワーク切断を検知しても、自動的に再接続を完了したり、以前の認証状態を復元したり、中断されたリクエストを自動再送したりはしない
- ただし `teacher_auth_token` が依然として有効であれば、ネットワーク復旧後、teacher は login_id / password を再度手動入力せずにリクエストを再試行できる
- マスターアプリ再起動有無、接続トークン変更有無、ローカルネットワーク IP 変更有無により、既存入力値がもはや有効でない場合がある
- したがって再接続後には必要に応じて、ユーザーがサーバーアドレス、接続トークン、teacher login_id / password を再確認または再入力するフローを許容する

現在のバージョンで実装しない項目:

- 自動再接続
- ネットワーク復旧検知後の自動認証再試行
- 未送信リクエストキュー保存および自動再送
- WiFi 再接続後のサーバー IP 自動再探索
- トークン無効化状況に対する自動復旧処理

これらの項目は後続バージョンのネットワーク安定性改善範囲として検討する。

## 7. 接続トークンおよび teacher 認証トークン規則

### 7-1. 基本概念

- マスターアプリは実行時に **ランダム接続トークン** を新しく生成する
- この接続トークンは **teacher が初回ログインリクエストを送る際に使用するサーバー接続用認証値** である
- teacher はマスターサーバーアドレスを知っていても、**有効な接続トークンなしではログインリクエストを通過できない**
- teacher は **IP + 接続トークン + teacher login_id + password** で初回ログインする
- ログイン成功時にサーバーは **`teacher_auth_token`** を発行する
- `teacher_auth_token` は以後の teacher 関連 API リクエスト認証に使用する

### 7-2. トークン役割区分

- **接続トークン**: サーバー接続およびログイン開始用
- **teacher_auth_token**: ログイン後の teacher 機能使用用

つまり、接続トークンはマスターサーバーに初めて接続してログインする時だけ使用する。  
出席処理、チェックイン、チェックアウト、管理機能リクエストなど、ログイン後の teacher 機能には接続トークンを再使用しない。  
以後のリクエストは、ログイン済み teacher 端末の認証状態を基準に処理する。

### 7-3. teacher_auth_token 維持および検証原則

- サーバーは **Cookie ベースセッションを維持しない**
- 代わりにログイン成功後に発行された `teacher_auth_token` を基準に teacher のログイン状態を維持する
- teacher 関連機能は **すでにログインされた teacher 端末** でのみ実行可能である
- 出席処理、チェックイン、チェックアウト、管理機能リクエスト時に **接続トークンは再要求しない**
- クライアントアプリは `teacher_auth_token` をローカルセキュアストレージに保存できる
- ネットワークが一時的に切断された後に再接続されても、`teacher_auth_token` が有効なら再ログインしなくてもよい

### 7-4. teacher_auth_token 満了および無効化規則

- `teacher_auth_token` の有効期間は **発行時点のマスターサーバーローカル日付まで** とする
- つまり、**マスターサーバーローカル日付が変更されると既存の `teacher_auth_token` は無効化される**
- 日付が変更された後の **最初の teacher API リクエスト時にサーバーがトークンを再検証** し、有効日付を過ぎていれば再ログイン必要応答を返す
- また、次の場合にも `teacher_auth_token` は無効化される
  - teacher ログアウト
  - teacher パスワード変更
  - `teachers.is_active = false`
  - マスターサーバー再起動
- 後続バージョンでは、日付基準満了の代わりに、より精密な満了ポリシーや更新方式へ改善できる

### 7-5. 再ログイン原則

- `teacher_auth_token` が無効化された場合、teacher は再度ログインしなければならない
- 再ログインは次の方式のいずれかで実行できる
  1. teacher が直接 `IP + 接続トークン + login_id + password` を手動入力
  2. 端末に保存されたサーバー情報、接続トークン、teacher アカウント情報を再使用して再ログイン
  3. 別のログイン済み teacher 端末で表示中のサーバー接続 QR をスキャンし、サーバーアドレスと接続トークンを再入力した後ログイン
- teacher 端末は再ログイン便宜のため **サーバー IP、接続トークン、teacher login_id、password またはそれに準ずるログイン再利用情報** をローカルセキュアストレージに保存できる

例示フロー:

```text
マスターアプリ実行
→ ランダム接続トークン生成
→ 管理者画面またはサーバー接続 QR にトークン表示
→ teacher が IP 入力
→ 接続トークン入力
→ teacher login_id / password 入力
→ ログイン成功
→ teacher_auth_token 発行
→ 以後、teacher API リクエストごとに Authorization ヘッダーで送信
```

## 8. クライアントアプリサーバー接続方式

マスターサーバーへ接続する方法は **teacher 基準で** 二つに定義する。

### 8-1. Private IP 直接入力方式

- teacher はマスターサーバーが実行中の機器の **ローカルネットワーク Private IP** を直接入力して接続できる
- 同じ機器で接続する場合には `127.0.0.1:40343` を使用できる
- teacher の初回ログインは **必ず IP 入力 + 接続トークン入力 + teacher login_id + password 入力** 手続きを経なければならない
- ログイン成功後は以後 teacher 関連機能を使用する際に **接続トークンを再使用しない**
- ログイン後の teacher 機能は `teacher_auth_token` を基準に処理する
- WiFi が切れた後に再接続された場合にも自動復旧を保証せず、以後のリクエストは接続可能状態およびログイン状態有効性を確認した上で処理する

例:

```text
192.168.x.x:40343
```

直接入力方式のフロー:

```text
teacher クライアント実行
→ IP 入力
→ 接続トークン入力
→ teacher login_id / password 入力
→ ログイン成功
→ teacher_auth_token 発行
→ 以後、teacher 機能使用
```

### 8-2. QR コードスキャン方式

#### 基本説明

- より簡便な接続のため **QR コードベース自動入力方式** を提供する
- QR コードは **マスターアプリ画面または既にログイン済みの teacher クライアント画面** で確認できる
- ただし、マスターアプリがバックグラウンド実行状態の場合には、必要に応じて前面 UI を再度開き、サーバー接続 QR を確認できる
- サーバー接続 QR には **マスターサーバー Private IP、ポート、接続トークン情報** を含む
- つまり、teacher は QR をスキャンすることで **IP と接続トークンを自動入力** できる
- ただし、**teacher login_id / password までは QR に含めない**
- WiFi 再接続後、既存のサーバーアドレスまたは接続トークンがもはや有効でない場合があり、現在バージョンではこれを自動復旧する fallback ロジックを持たない

サーバー接続 QR には次の情報が含まれる。

- マスターサーバー Private IP
- マスターサーバーポート (`40343`)
- マスターアプリ実行時に生成された接続トークン

例示構造:

```text
SERVER:http://192.168.x.x:40343
TOKEN:random-generated-token
```

QR コード使用フロー:

```text
新しい teacher クライアント実行
→ サーバー接続 QR スキャン
→ マスターサーバーアドレスおよび接続トークン自動入力
→ teacher login_id / password 入力
→ ログイン成功
→ teacher_auth_token 発行
→ 以後、teacher 機能使用
```

この方式の目的:

- ネットワークアドレスと接続トークンをそれぞれ手動入力する必要を減らす
- 新しい teacher 機器が **一度のスキャンで IP と接続トークンを確保** できる
- 接続トークンは初回ログインまでのみ使用し、以後の機能には再使用しない

### 8-3. 学生側接続原則

- student は **マスターサーバーの直接接続主体ではない**
- student は普段 **マスターと接続されていなくてもよい**
- student は **別途会員登録 UI を使用しない**
- student アカウントは teacher または admin がマスター側で登録することを基本とする
- student 端末では **初回ログイン 1回と初回 QR 生成 1回のみ** マスターと通信可能状態が必要である
- 初回ログインと初回 QR 生成が終わった後は、基本的な QR 表示使用のために毎回マスター接続手続きを繰り返さない

## 9. 管理者アクセス方式

- 管理機能は **クライアントアプリの管理モード** を通じてアクセスすることを基本原則とする
- teacher 関連リクエストは **Authorization ヘッダーの `teacher_auth_token` 検証** を経なければならない
- つまり、管理者は別途ブラウザ URL を直接入力する方式より、**teacher クライアントでログイン後、管理モードへ進入する** フローを基本 UX とする
