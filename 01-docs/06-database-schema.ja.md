# 04. QR / 出席 / 勤怠ポリシー

## 1. 文書の目的

この文書は、**QRの種類、表示/スキャンモード、出席QR生成ルール、teacherの出勤/退勤ルール、studentの出席ルール、出席noteルール**を整理した基準文書である。  
サーバー接続用認証とトークン詳細は `02-runtime-architecture-and-connection.md`、権限/UI遷移構造は `03-role-permission-and-client-flow.md`、保存構造は `05-domain-rules-and-schema.md` で詳細に定義する。

---

## 2. QRの種類の区分

**何のためのQRか（用途）** と **どのように使うか（使用モード）** を分けて説明する。

要点は次のとおりである。

- **サーバー接続QR**: 新しいteacherクライアントアプリを **マスターサーバーに接続し、初回ログインに必要なサーバー情報と接続トークンを確保** するためのQR
- **出席QR**: studentまたはteacherの **アカウント情報を表示して出席処理** を行うためのQR
- **表示モード**: 自分のQRを画面に表示するモード
- **スキャンモード**: 他人のQRを読み取るモード

つまり、**サーバー接続QR / 出席QRは用途の区分** であり、**表示モード / スキャンモードは使用方式の区分** である。

---

## 3. 用途の区分

### 3-1. サーバー接続QR

- 目的: **teacherクライアントがマスターサーバーのアドレスを受け取り、初回ログインに必要なサーバー情報と接続トークンを確保** するためのQR
- 含まれる情報: マスターサーバーのPrivate IP + ポート (`40343`) + 実行時に生成された接続トークン
- 表示位置: **すでにログイン済みのteacherクライアント、またはマスターアプリ画面**

例示構造:

```text
SERVER:http://192.168.x.x:40343
TOKEN:random-generated-token
````

実際の使用シナリオ例:

```text
状況: 新しいタブレット1台を先生用端末として塾に追加する

1. マスターアプリが起動し、ランダム接続トークンが生成される
2. マスターアプリ画面、またはすでにログイン済みのteacherアプリ画面でサーバー接続QRを表示する
3. 新しいタブレットでteacherクライアントを起動する
4. 新しいタブレットがサーバー接続QRをスキャンする
5. アプリがマスターサーバーのアドレスと接続トークンを自動入力する
6. teacherが login_id / password を入力してログインする
7. ログイン成功後、teacher_auth_token が発行され、機能を利用できる
```

このQRは **出席処理用ではなく、teacher端末のサーバーアドレス + 接続トークン入力用** である。

### 3-2. 出席QR

#### 基本概念

* 目的: **studentまたはteacherのアカウント情報をQRで表示し、出席処理** を行うためのQR
* teacher QRも本人のアカウント情報をQRとして表示できる
* 出席QRはサーバー接続QRとは別概念であり、役割を混同しないようUI上で明確に区分する必要がある
* student QRとteacher QRの区分は別途typeフィールドを使わず、**code prefix体系** で処理する
* たとえば `student_code` は `S` で始まり、`teacher_code` は `T` で始まる

#### 出席QR生成形式

出席QRは次の形式を使用する。

```text
LOGIN_ID|CODE|TIME_SLOT|PORT
```

例:

```text
kim01|S0001|29384721|40343
park02|T0003|29384721|40343
```

構成要素の意味:

* `LOGIN_ID`: 当該アカウントの login_id
* `CODE`: student_code または teacher_code
* `TIME_SLOT`: 60秒単位の再生成基準値
* `PORT`: 現在アプリが使用しているポート番号

#### student出席QR生成原則

* student端末は初回ログイン時にQR生成に必要なローカル保存情報を確保する
* 以後、student出席QRは **オフラインでローカル生成** できる
* student出席QRは **`login_id + student_code + time_slot + port`** 構造で生成する
* `TIME_SLOT` は **60秒単位の再生成基準値** として使用する
* したがってstudentアプリは、同じネットワークに継続して接続されていなくてもQRを継続的に再生成できる

#### teacher出席QR生成原則

* teacher端末は初回ログイン時に teacher_code およびQR生成に必要なローカル保存情報を確保する
* 以後、teacher出席QRは **現在のログイン状態と無関係に** ローカルで生成できる
* teacher出席QRは **`login_id + teacher_code + time_slot + port`** 構造で生成する
* `TIME_SLOT` は **60秒単位の再生成基準値** として使用する
* したがってteacherも、現在 `teacher_auth_token` が失効している状態であっても、過去に必要情報が保存されていれば本人QRを継続表示できる

#### 検証およびセキュリティルール

* スキャナーは `CODE` prefix を基準にstudent QRとteacher QRを区分する
* 出席QRの再生成周期は **60秒単位** とする
* サーバーは **現在のマスターサーバーのローカル time_slot とQRの time_slot が正確に一致する場合のみ有効** として扱う
* サーバーはQR検証時に次の順序に従う

1. QRをパースする
2. code prefix で student / teacher を区分する
3. 当該テーブルで code を照会する
4. row の `login_id` とQRの `login_id` が一致するか確認する
5. QRの `time_slot` と現在のマスターサーバー `time_slot` が一致するか確認する
6. `port` 値が現在のサーバーポートと一致するか確認する
7. すべて一致すれば有効として処理する

* 現在のQR生成ルールは **MVP 1次向けの単純検証構造** であり、強い偽造防止を保証する秘密署名構造ではない
* ポート番号はQR生成ルールの構成要素として使用するが、強い秘密値とは見なさない
* 後続バージョンでは、seed、HMAC、署名ベース方式によるQR偽造防止強化を検討できる

#### 学生アカウントおよび端末原則

* 学生は別途の会員登録UIを使用しない
* 学生アカウントはteacherまたはadminが登録する
* 学生端末では **初回ログイン1回と初回QR生成1回** のみマスターと通信可能な状態が必要である
* 初回ログインと初回QR生成以後は、学生が同じネットワークに継続して接続されていなくても基本的なQR表示利用は可能である
* 出席QR自体にはWiFi情報を含めない

#### teacher端末使用原則

* teacher QR表示は、過去に一度ログインして必要なローカル保存情報を確保した端末であれば、現在のログイン状態と無関係に可能である
* 一方で、出席処理、チェックイン、チェックアウトのためのスキャンモードは **ログイン済みのteacher端末** でのみ可能である
* つまり、QRを見せることと、QRを読み取ってサーバーに記録することは区分する
* 出席処理、チェックイン、チェックアウト時に **接続トークンは不要** である
* 接続トークンはteacher端末がマスターサーバーに初回ログインするときにのみ使用する
* `teacher_auth_token` が無効化された場合、teacherは再ログイン後にスキャンモードとサーバー記録機能を利用できる

実際の使用シナリオ例1: 学生アカウント登録および初回QR生成

```text
状況: 新しい学生をシステムに登録する

1. teacherまたはadminがマスターで学生アカウントを登録する
2. 学生端末で初回ログインを1回実行する
3. マスターと通信して初回QR生成に必要な情報を保存する
4. 以後、学生はオフラインでも本人QR画面を継続利用できる
```

実際の使用シナリオ例2: 学生出席

```text
状況: 学生が塾に到着する

1. 学生が自分のアプリで本人QRを画面に表示する
2. ログイン済みteacher端末がスキャンモードで学生QRを読み取る
3. システムが login_id, student_code, time_slot, port を検証する
4. 有効なら学生出席が記録される
```

実際の使用シナリオ例3: 先生の出勤

```text
状況: 先生が出勤する

1. 先生が本人アプリで本人QRを画面に表示する
2. 別のログイン済みteacher端末のスキャンモードがそのQRを読み取る
3. システムが login_id, teacher_code, time_slot, port を検証する
4. 同じ日の最初のスキャンであれば出勤として処理する
```

実際の使用シナリオ例4: 先生の退勤

```text
状況: 先生が退勤する

1. 先生が自分のアプリで再び本人QRを画面に表示する
2. 別のログイン済みteacher端末のスキャンモードがそのQRを読み取る
3. システムが同じ日の既存出勤記録を確認する
4. 2回目のスキャンであれば退勤として処理する
```

つまり、出席QRは **サーバー接続用ではなく、人の出席処理用** である。

## 4. 使用モードの区分

### 4-1. 表示モード

* **studentとteacherの両方が使用可能**
* 表示モードは **自分のQRを画面に表示するモード** である
* studentは表示モードで **本人の出席用QR** を画面に表示する
* teacherも表示モードで **本人QR** を画面に表示できる
* teacher QRは他のログイン済みteacher端末のスキャンモードで読み取られ、**出勤/退勤処理** に利用できる

実際の使用シナリオ例:

```text
学生アプリを起動
→ 本人QRを表示
→ teacherがスキャン
→ 学生出席を処理
```

または

```text
teacherアプリを起動
→ 本人QRを表示
→ 別のログイン済みteacher端末のスキャンモードが読み取る
→ teacherの出勤または退勤を処理
```

### 4-2. スキャンモード

* **スキャンモードはteacherのみが使用できる機能** である
* スキャンモードは **他人のQRを読み取るモード** である
* スキャンモードが有効化されたteacherアプリでQRをスキャンすると出席処理が行われる
* student QRをスキャンするとstudent出席処理が実行される
* teacher QRをスキャンするとteacher出勤/退勤処理が実行される
* スキャンモードは **ログイン済みteacher端末でのみ使用可能** であり、`teacher_auth_token` が無効であれば使用できない

実際の使用シナリオ例:

```text
teacherがスキャンモードを起動
→ 学生QRをスキャン
→ 学生出席を記録
```

または

```text
teacherがスキャンモードを起動
→ teacher QRをスキャン
→ teacherの出勤または退勤を記録
```

## 5. 混同しないために一文で整理

```text
サーバー接続QR = 端末をサーバーに接続し、初回ログインに必要なサーバー情報と接続トークンを確保するためのQR
出席QR = 人の出席を処理するためのQR
表示モード = 自分のQRを見せるモード
スキャンモード = 他人のQRを読み取るモード
```

## 6. 出席処理方式

### 6-1. 共通原則

* 現場ではQRコードなどを用いて出席処理が可能
* 出席機能もクライアントアプリ内で提供する
* QR関連機能は **表示モード** と **スキャンモード** に区分する
* **表示モード** はstudentとteacherの両方が使用でき、本人アカウントQRを画面に表示する用途である
* **スキャンモード** はログイン済みteacher端末でのみ使用でき、スキャンモードが有効化されたアプリでQRを読み取ると出席処理が行われる
* スキャンモードでstudent QRを読み取るとstudent出席処理を行う
* スキャンモードでteacher QRを読み取るとteacher出勤/退勤処理を行う
* 出席処理、チェックイン、チェックアウト時に **接続トークンは再利用しない**

### 6-2. 先生の出勤/退勤ルール

* teacher QRスキャンルールは **同じ日の最初のスキャンは出勤**、**2回目のスキャンは退勤**、**3回目以降はエラー処理** と定義する
* つまり、`teacher_attendance` は **1日最大2回のスキャンのみ許可** する
* teacher QRの最初のスキャン時に `teacher_attendance` row を生成し、2回目のスキャン時に同じrowの `checkout_at` を更新する
* 出勤だけ記録して退勤を記録しないまま日付が変わった場合、翌日の最初のスキャンは **新しい出勤処理** として開始する
* 前日の退勤未記録件は維持し、以後UIで **退勤未記録エラー (`checkout_at IS NULL`)** として表示できるようにする

### 6-3. 学生出席ルール

* student出席は重複入力を許可するが、**同じ学生は5分以内に再スキャンできないよう制限** して多重入力を防止する
* 学生が複数回出席できる理由は、連続しない時間帯に複数の授業を受ける可能性があるためである
* ただしこのルールはMVP 1次の単純な保護ルールであり、以後のバージョンでは授業/時間ベースでもっと精緻に改善する余地がある

### 6-4. 学生端末接続原則

* 学生は別途の会員登録UIを使用しない
* 学生アカウントはteacherまたはadminが登録する
* 学生端末では初回ログイン1回と初回QR生成1回のみマスターと通信可能な状態が必要である
* 初回QR生成以後は、学生が同じネットワークに継続して接続されていなくても基本的なQR表示利用は可能である

## 7. 出席記録ルール

### 7-1. 基本保存原則

* MVP 1次で `teacher_attendance` と `student_attendance` は **記録そのものを保存し、状態は解釈して表示する方式** を使用する
* したがってattendanceテーブルには `status` と `reason` を置かず、必要な場合のみ手動入力用 `note` を使用する
* `teacher_attendance.note`, `student_attendance.note` は基本的にNULL許容
* noteは特殊ケースがあったかどうかを簡潔に手動記入するための用途である

### 7-2. note作成 / 修正権限

* 出席note関連の作成者IDは **原本作成者** ではなく、**現在値の最終修正者** を意味する
* したがって `note_editor_teacher_id` は現在のnote値を最後に修正したteacher IDを保存する
* 原本作成者を確認するには `audit_logs` を参照する必要がある
* non-admin teacherも出席処理過程でnote作成は可能である
* non-admin teacherは自分が最後に修正した出席noteのみ再修正可能であり、他人が最後に修正したnoteは修正できない
* adminは出席note修正に制限を設けない

### 7-3. 先生の出勤/退勤解釈ルール

* teacherの出勤/退勤状態は保存せず、`checkin_at`, `checkout_at` の値を基準に解釈する
* 同じ日の最初のスキャンは出勤、2回目のスキャンは退勤、3回目以降はエラーとして処理する
* つまり、`teacher_attendance` は1日最大2回のスキャンのみ許可する
* teacher QRの最初のスキャンは `teacher_attendance` row 生成、2回目のスキャンは同じrowの `checkout_at` 更新として処理する
* 出勤だけ記録して退勤を記録しないまま日付が変わった場合、翌日の最初のスキャンは新しい出勤として処理する
* 前日の未完了rowは維持し、以後UIで退勤未記録状態として解釈・表示できるようにする

### 7-4. 学生出席ルール

* `student_attendance` はMVP 1次段階で **授業別出席ではなく、学生単位のスキャン記録** を保存する構造を使用する
* student QRスキャンでは `class_id` を使用しない
* student出席rowは `student_id` と `scanned_at` を中心に保存する
* student出席の日付判定は `scanned_at` のマスターサーバーローカル日付基準で処理する
* 学生は重複出席が可能だが、同じ学生基準で5分以内に再スキャンできないよう制限して多重入力を防止する
* 学生が複数回出席できる理由は、連続しない時間帯に複数の授業を受ける可能性があるためである
* このルールはMVP 1次の単純な保護ルールであり、必要に応じて後続バージョンで授業/時間ベース政策としてさらに精緻化できる
