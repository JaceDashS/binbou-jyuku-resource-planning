# 01. 製品範囲 / MVP 方針

## 1. 文書目的

この文書は、**小規模学習塾統合管理システムの背景、目標、MVP 範囲、除外範囲、優先順位**を整理した基準文書である。  
接続方式、権限、QR / 出席ポリシー、DB スキーマはそれぞれ別文書で詳細を定義する。

---

## 2. 背景

- 対象: 施設・予算が不足している小規模個人塾
- 制約:
  - 商用 ERP を購読する予算が不足している
  - 別途サーバーを継続的に維持・保守する予算が不足している
- 目標:
  - **本当に必要な機能だけを最小実装**し、運営効率を高めること
  - 一つのシステムの中で管理 / 出欠 / 運営機能を統合すること
  - システムの **最大要求性能をミニ PC レベルに制限**し、高性能サーバーや高価な機器を前提にしないこと
  - 非専門家でも各機能の目的を理解しやすい **用語と表現** を使用すること

---

## 3. 基本方針

### 3-1. システム構成方針

- **過剰な汎用 ERP ではなく、小規模塾運営に必要な最小機能中心のシステム**
- システムは **マスターアプリ + クライアントアプリ** 構成で運用
- **クライアントアプリは React Native ベースのクロスプラットフォームアプリ** として開発
- マスターアプリは SQLite と Go ベースのマスターサーバーを含む実行型アプリとして提供
- マスターアプリは前面 UI として実行される場合もあれば、必要に応じて **バックグラウンドのサーバープロセス形態** で実行される場合もある

### 3-2. 運営および役割原則

- 小規模施設を前提とするため、**teacher と staff を別役割として分離しない**
  - その理由は **最小実装達成** と **UX 単純化** を優先するためであり、運営補助役割も teacher 範囲で処理する

---

## 4. 主要機能要求事項（範囲基準）

### 4-1. 運営管理

- teacher 給与管理
  - ただし **payroll ページは admin のみアクセス可能**
- 保護者または student 問い合わせ管理
- 授業開設 / 変更 / 閉講管理
- teacher 出退勤管理
- アカウント / 個人情報修正ページ
  - teacher は **自分のアカウントの** `login_id`, `password`, `name`, `email`, `phone`, `emergency_contact` を修正できる
  - student アカウント情報は **teacher 画面を通じてのみ** 修正可能であり、修正範囲は `login_id`, `password`, `name`, `email`, `guardian_email`, `phone`, `guardian_phone` に限定する
  - MVP 1次では **student 本人が直接個人情報を修正しない**
  - admin アカウントでログインした場合は、`is_active`, `is_admin` などの状態 / 権限変更ページへ入るボタンが表示される

### 4-2. 学生管理

- 子どもの出席（出席記録）管理
- 宿題遂行率管理
  - ただし **MVP 1次バージョンでは実装しない**
  - 初期運営安定化後、後続バージョンで設計および実装予定
- 学生管理画面では **学生データと関連 teacher 情報** を一緒に照会できなければならない
- ただし MVP 1次で学生管理領域から直接修正可能な範囲は **問い合わせ** と **出席関連記録（note 含む）** に限定する

### 4-3. 監査ログ / 変更履歴

システムの主要データ修正は **変更履歴（audit log）** として記録されなければならない。

目的:

- 誰がデータを修正したのか追跡
- いつ、どのような変更が発生したのか記録
- 問題発生時に過去状態を復元または分析可能にする

詳細な保存構造、権限、スキーマは `05-domain-rules.ja.md` と `06-database-schema.ja.md` で定義する。

## 5. MVP 1次範囲と除外範囲

### 5-1. MVP 1次で含まれる方向性

- 最小機能優先
- マスターアプリ + クライアントアプリ構成
- ローカル中心運用
- 変更履歴追跡可能
- SQLite ローカル DB + Go サーバー基盤
- teacher / student 中心の単純なアカウントモデル
- QR ベース出席 / 出退勤
- 問い合わせ、授業、学生 / teacher、給与、出席の最小管理機能
- teacher 初回ログイン後 `teacher_auth_token` ベース機能使用
- student / teacher 出席 QR のローカル生成およびオフライン使用

### 5-2. MVP 1次で除外する項目

#### 機能除外

- **宿題遂行率管理**
- **student 直接個人情報修正機能**
- **student 会員登録 UI**
- **student のマスターサーバー直接接続主体化**
- **授業別の精密な出席ポリシー**
- **メール統合ビュー**
- **機械学習機能**
- **QR 偽造防止セキュリティ高度化（HMAC、署名など）**
- **ネットワークセキュリティ強化およびファイアウォール / ポリシー対応自動化**
- **WiFi 切り替え連携 QR 接続フロー**
- **WiFi 再接続自動 fallback / 自動復旧**
- **ポート fallback 自動切り替え**
- **ポート変更時の QR 再同期自動処理**
- **退勤未記録専用高度化 UI**
- **公式マスターマイグレーション機能**
- **CSV export**
- **クラウド DB / 外部サーバー DB 導入**

#### 運営範囲除外

- 高性能サーバーや高価な機器前提
- 複雑な汎用 ERP 水準の機能
- 継続的なサーバー運用インフラ

## 6. 後続で検討可能な拡張機能

### 6-1. 機能拡張

- **student 直接個人情報修正機能の拡張可能性**
  - 後続バージョンでは student アプリで直接個人情報修正機能または修正リクエスト機能を検討できる
  - 現在構造では students テーブル直接修正方式より、**修正リクエスト作成 → teacher 承認 → 実際反映** 構造を優先検討する
- **宿題遂行率管理機能**
  - 初期運営安定化後、後続バージョンで設計および実装を検討できる
- 出欠、宿題率、担当 teacher 情報などを総合した **機械学習機能追加**
  - 例: 欠席可能性予測
  - 例: 宿題未提出リスク student 探知
  - 例: 管理優先順位推薦
  - ただし、目標とする PC スペックで実装不可能な場合は除外する
- **メール統合ビュー機能追加可能性**
  - 塾運営関連メールを一画面で統合照会できる機能を後続で検討できる
  - 実際の必要性が確認された場合、後続バージョンで設計および追加可能
- **student 出席ポリシー高度化可能性**
  - 現在は student 単位スキャン記録と 5分再スキャン制限のみ適用するが、後続バージョンでは授業 / 時間ベースでより精密に改善できる

### 6-2. セキュリティ / ネットワーク / 運営拡張

- **QR 偽造防止セキュリティ高度化可能性**
  - 現在の QR 生成規則は MVP 1次向けの単純検証構造であり、強い偽造防止を保証する秘密署名構造ではない
  - 後続バージョンでは seed、HMAC、署名ベース方式による QR 検証強化を検討できる

- **teacher 認証トークン運用高度化可能性**
  - MVP 1次では teacher ログイン成功後 `teacher_auth_token` を発行し、以後 teacher 関連 API リクエストは Authorization ヘッダーベースで処理する
  - `teacher_auth_token` は発行時点のマスターサーバーローカル日付まで有効であり、日付変更時に無効化される
  - 後続バージョンでは、より精密な満了ポリシー、更新方式、refresh token 構造などを検討できる

- **ネットワーク接続安定性およびセキュリティ強化可能性**
  - 現在構造では WiFi が切断された後に再接続される状況に対して、アプリが自動で接続状態を復旧したり認証値を再交渉する fallback ロジックを実装しない
  - ただし `teacher_auth_token` が有効であれば、ネットワーク復旧後に再度 login_id / password を手動入力せずリクエストを再試行できる
  - ファイアウォール、ローカルネットワークポリシー、OS セキュリティ設定による接続問題対応は今後の改善項目として検討できる
  - 外部から入る信号に対するセキュリティ強化や追加防御ロジックも後続バージョンで検討できる

- **WiFi 切り替え連携 QR 接続フロー検討可能性**
  - WiFi SSID または WiFi パスワードを QR コードに含め、デフォルトカメラを通じて WiFi 切り替えと同時にアプリを実行する流れが実現可能かは後続で検討できる

- **ポート衝突対応高度化可能性**
  - 現在は `40343` 固定ポートのみ使用し、衝突時には実行を中断する
  - 後続バージョンでは fallback ポート自動切り替えまたは代替ポート探索機能を検討できる

- **ポート変更対応高度化可能性**
  - 現在、QR 生成規則にはポート番号が含まれるため、ポート変更後の端末再同期ポリシーは後続バージョンで検討できる
  - ただし MVP 1次ではポート変更時の自動再同期または自動復旧は扱わない

- **退勤未記録 UI 表示強化可能性**
  - 現在も退勤未記録状態を表示できるが、後続バージョンでは専用警告 UI、フィルタ、管理者確認フローなどによりより明確に高度化できる

### 6-3. 運営便利機能

- マスターアプリマイグレーション機能
- CSV export

## 7. 現在のアイデアに基づくシステム目標

> **小規模個人塾向けの、サーバー費用と購読費用を最小化したローカル中心のクロスプラットフォーム統合運営システム**

核心は次の 4点である。

- 最小機能優先
- マスターアプリ + クライアントアプリ構成
- ローカル中心運用
- 変更履歴追跡可能

---

## 8. 優先順位草案

### 8-1. 1次必須

- 管理者ログイン / 権限区分
- teacher 管理
- student 管理
- 出席管理
- 授業開設 / 変更 / 閉講管理
- 保護者または student 問い合わせ管理
- 出退勤管理
- 変更履歴記録

### 8-2. 2次必須

- 給与管理
- QR ベース出席処理
- ローカルサーバー + 端末役割分離 / 初期設定

### 8-3. 3次拡張

- 宿題率分析
- teacher / student 総合指標
- 機械学習ベース予測機能
- メール統合ビュー機能可能性検討
